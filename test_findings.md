## 🔴 Critical 级别缺陷 (P0)

### BUG-001: Supabase PostgreSQL 数据库连接失败
- **严重程度**: 🔴 Critical (P0)
- **影响范围**: 所有需要数据库操作的API（认证、关键词、热点等）
- **复现步骤**:
  1. 启动后端服务
  2. 尝试调用任何写操作API（如注册、登录）
  3. 观察返回 HTTP 500 错误
- **实际结果**: 
  ```json
  {
    "error": "注册失败，请稍后重试"
  }
  ```
  服务器日志: `Error: P1001: Can't reach database server at db.kipjyvaiebzzpycvhmzz.supabase.co:5432`
- **预期结果**: 应成功连接数据库并完成操作
- **根本原因**: 
  - Supabase 数据库实例可能已暂停（免费版特性）
  - 或网络连接问题
  - 或数据库凭据过期/变更
- **影响用户**: **100%的用户无法使用系统**
- **修复建议**:
  1. **立即**: 检查 Supabase 控制面板确认数据库状态
  2. **短期**: 添加数据库连接健康检查到 `/api/health` 端点
  3. **中期**: 实现本地 SQLite 作为 fallback 方案
  4. **长期**: 配置数据库连接池和自动重连机制

## 🟠 Major 级别缺陷 (P1)

### BUG-002: 用户注册功能不可用
- **严重程度**: 🟠 Major (P1)
- **影响模块**: 用户认证系统
- **前置依赖**: BUG-001
- **HTTP状态码**: 500 Internal Server Error (应为201)
- **测试数据**: 
  ```
  POST /api/auth/register
  Body: {email: "test@example.com", password: "Test123456"}
  ```
- **实际行为**: 返回通用错误信息 `"注册失败，请稍后重试"`
- **预期行为**: 
  - 成功时返回 201 Created
  - 包含用户信息和JWT Token
- **业务影响**: 新用户无法注册使用系统

### BUG-003: 用户登录功能不可用
- **严重程度**: 🟠 Major (P1)
- **影响模块**: 用户认证系统
- **前置依赖**: BUG-001
- **HTTP状态码**: 500 (应为200或401)
- **测试场景**:
  - 正确凭据登录 → 返回500 (应为200)
  - 错误凭据登录 → 返回500 (应为401)
- **安全风险**: ⚠️ 错误凭据应返回401而非500，避免泄露内部错误
- **业务影响**: 已有用户无法登录系统

### BUG-004: 忘记密码功能不可用
- **严重程度**: 🟠 Major (P1)
- **影响模块**: 密码恢复流程
- **前置依赖**: BUG-001
- **HTTP状态码**: 500 (应为200)
- **设计评价**: ✅ 采用防枚举设计（无论邮箱是否存在都返回相同响应）
- **业务影响**: 用户无法找回密码

### BUG-005: Token刷新机制失效
- **严重程度**: 🟠 Major (P1)
- **影响模块**: 会话管理
- **前置依赖**: BUG-002, BUG-003
- **根因**: 无法获取初始 RefreshToken
- **影响**: 
  - 用户会话无法延续
  - Access Token 过期后需重新登录
- **用户体验**: 🔴 严重影响（频繁要求重新登录）

## 🟡 Minor 级别缺陷 (P2)

### BUG-006: 限流策略过于严格
- **严重程度**: 🟡 Minor (P2)
- **影响模块**: 注册/登录API
- **观察到的限制**:
  - 注册接口: 3次/分钟
  - 登录接口: 可能类似限制
- **问题**: 在自动化测试中容易触发限流
- **建议**: 
  - 对IP地址进行分级限流
  - 提供更明确的限流提示（Retry-After头已实现✅）
  - 考虑增加验证码机制替代严格限流

### BUG-007: 错误信息不够具体
- **严重程度**: 🟡 Minor (P2)
- **示例**: 
  - 当前: `"注册失败，请稍后重试"`
  - 建议: 区分网络错误、数据库错误、验证错误等
- **用户体验**: 用户不知道如何解决问题
- **安全性考量**: 需平衡详细度与安全性（不暴露敏感信息）
